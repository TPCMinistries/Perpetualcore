**1a. `lib/rfp/feed.ts`** — server helpers:

```ts
export interface FeedFilters {
  org_id: string;
  sources?: string[];               // ['sam_gov', 'grants_gov', ...]
  deadline_within_days?: 7 | 30 | null;
  min_amount?: number | null;
  cursor?: { fit_score: number; opp_id: string } | null;  // keyset pagination
  limit?: number;                   // default 25
}

export interface FeedRow {
  opp_id: string;
  source: string;
  title: string;
  agency: string | null;
  amount_min: number | null;
  amount_max: number | null;
  deadline: string | null;
  brief: string | null;
  url: string | null;
  fit_score: number;
  tier: 'Strong fit' | 'Good fit' | 'Marginal' | 'Weak';
  chips: string[];
  summary: string | null;
  needs_review: boolean;
}

export async function buildFeedQuery(filters: FeedFilters): Promise<{ rows: FeedRow[]; next_cursor: FeedRow['opp_id'] | null }>
```

Implementation:
  - **CRITICAL**: Use the request-scoped `createClient` from `@/lib/supabase/server` (the one that reads cookies via `next/headers`). Do NOT use `createAdminClient` — that bypasses RLS and would let any authenticated user see any org's matches. RLS on rfp_opp_matches is "member-only" — selecting org_id = filters.org_id naturally returns rows ONLY because the caller is a member, AND we include the explicit `.eq('org_id', filters.org_id)` for query-plan + safety belt.
  - Join via Supabase relational select: `select('opp_id, fit_score, chips, summary, rfp_opportunities ( source, title, agency, amount_min, amount_max, deadline, brief, url, needs_review )')`.
  - Apply filters:
    - sources: `.in('rfp_opportunities.source', filters.sources)` (use the related-table filter syntax — confirm with current Supabase JS docs; if not supported, fetch unfiltered then filter in-memory at this scale).
    - deadline_within_days: `.lte('rfp_opportunities.deadline', now + N days)` AND `.gte('deadline', now)` (don't show already-past deadlines).
    - min_amount: `.gte('rfp_opportunities.amount_max', min_amount)`.
  - Order: `fit_score desc, opp_id asc` (opp_id is the tiebreaker for stable keyset pagination).
  - Cursor: when `filters.cursor` provided, add `.or('fit_score.lt.{cursor.fit_score},and(fit_score.eq.{cursor.fit_score},opp_id.gt.{cursor.opp_id})')`.
  - Limit: `.limit(filters.limit ?? 25)`.
  - Map results into FeedRow[]; compute `tier` from fit_score using `tierFor()` from lib/rfp/scoring/weights.ts.
  - Return `{ rows, next_cursor }` where next_cursor is `rows[rows.length-1].opp_id` if `rows.length === limit`, else null.

**1b. `app/api/rfp/opps/route.ts`** — GET handler:
  - **Use the request-scoped server `createClient` (cookies-bound) when calling buildFeedQuery indirectly — buildFeedQuery itself constructs the client, but the route must run with full cookie context (no admin bypass).**
  - Parse query params via Zod schema. Coerce sources from comma-separated string. Coerce min_amount to number.
  - org_id is required; if absent, 400. Membership is enforced by RLS — non-member just gets empty rows, which is fine.
  - Cursor: accept as base64-encoded JSON in `?cursor=` to keep URLs clean.
  - Call buildFeedQuery, return `{ rows, next_cursor }`.
  - On error, return 500 with `{ error: 'feed_unavailable' }` (no stack leak).

**1c. `app/api/rfp/opps/[id]/route.ts`** — GET single opp:
  - Same RLS rule: request-scoped `createClient`, NOT admin. The single-row read also depends on the user being an org member to see the rfp_opp_matches row.
  - Params: `id` (opportunity id), and `org_id` from query string (required, lets us look up the right rfp_opp_matches row).
  - Returns the same FeedRow shape plus a few extra fields: `score_breakdown` (jsonb), `posted_at`, `keywords`, `geo`, `raw_json` (for power-users).
