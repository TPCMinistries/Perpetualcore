Update `FeedRow`:
```ts
export interface FeedRow {
  // ... existing fields ...
  scored_for_org_id: string;
  scored_for_org_name: string;
  scored_for_org_type: 'nonprofit' | 'forprofit' | 'dual';
}
```

Update `buildFeedQuery`:
  - When `dual_org_ids` is provided (non-empty), query `rfp_opp_matches` with `.in('org_id', dual_org_ids)` instead of `.eq('org_id', org_id)`.
  - **Deduplicate by opp_id**: if the same opp scores for multiple orgs, keep only the row with the highest fit_score. Implement via a window function in Supabase OR by fetching with the raw .in and post-processing in JS:
    ```ts
    const byOpp = new Map<string, FeedRowRaw>();
    for (const r of rows) {
      const existing = byOpp.get(r.opp_id);
      if (!existing || r.fit_score > existing.fit_score) byOpp.set(r.opp_id, r);
    }
    ```
  - Join `rfp_orgs` to populate scored_for_org_name and scored_for_org_type. Supabase nested select syntax: `select('opp_id, fit_score, ..., rfp_orgs ( name, type ), rfp_opportunities ( ... )')`.
  - Pagination: keyset cursor still uses (fit_score, opp_id). Dedup happens after fetch — over-fetch by 2× to compensate (fetch limit*2, dedupe, take top `limit`). Document this in a comment.

**1b. Update `app/api/rfp/opps/route.ts`** GET handler:
  - Add Zod fields: `dual: z.coerce.boolean().optional()`, `mode: z.enum(['all','nonprofit','forprofit']).optional()`.
  - When `dual === true`:
    1. Confirm caller is a member of `org_id` AND that org has type='dual' (load via `getOrgForUser`).
    2. Use `listUserOrgs()` (from lib/rfp/orgs.ts) to get all the user's orgs.
    3. Filter to orgs matching the requested `mode`:
       - `mode='all'` (default): include both nonprofit and forprofit orgs the user owns. Skip 'dual' orgs themselves to avoid double-counting (the dual org is the lens, not a target).
       - `mode='nonprofit'`: only nonprofit type orgs.
       - `mode='forprofit'`: only forprofit type orgs.
    4. **Empty-set guard:** if `filteredOrgIds.length === 0` (user owns a dual lens but is NOT a member of any underlying nonprofit/forprofit orgs that match the mode), return early with `{ rows: [], next_cursor: null, empty_reason: 'no_member_orgs' }`. Do NOT call buildFeedQuery — there's nothing to query and the response shape needs the `empty_reason` discriminator so the client can render an actionable empty state. Same applies if `mode='nonprofit'` and the user has dual + forprofit only (no nonprofit member orgs) — return `empty_reason: 'no_member_orgs'` with the resolved mode in the response so the UI can be specific.
    5. Otherwise, pass `dual_org_ids = filteredOrgIds, mode_filter = mode` to buildFeedQuery.
  - When `dual !== true`: existing single-org behavior unchanged.

**Response shape extension** (only relevant for dual=true):
```ts
{ rows: FeedRow[], next_cursor: string | null, empty_reason?: 'no_member_orgs' }
```
The optional `empty_reason` field is set ONLY when rows is empty due to the user not being a member of any qualifying underlying orgs. It is omitted in all normal cases (rows populated, or rows empty due to filters). DiscoveryClient renders an empty-state asking the user to also join the underlying orgs when this discriminator appears.
