**2g. Count-anomaly check (orchestrator-side, NOT scraper-side):**

For each successful scrape (parsed_count >= 1):
  1. Read `baseline = await getRollingBaseline(source)`.
  2. If baseline !== null AND parsed_count < baseline * 0.5, call `recordDrift({ source, reason: 'count_anomaly', details: { parsed: parsed_count, baseline, drop_pct: 1 - parsed_count / baseline } })`. Do NOT skip the upsert — the parsed records ARE saved; the drift row is purely a signal that something may be wrong with the parser.
  3. Regardless of anomaly outcome, call `recordBaseline(source, parsed_count)` to update the rolling baseline. (We always record baselines on successful runs; that way the rolling window self-heals if a source legitimately shrinks long-term.)

Why this approach: `count_anomaly` is the simpler of the two options the checker proposed (count-anomaly vs partial_failure). It catches the "parser broke and now we're getting 2 records instead of 50" case without per-record schema introspection. Partial-failure-per-record can be added in Phase 10 admin tooling.
