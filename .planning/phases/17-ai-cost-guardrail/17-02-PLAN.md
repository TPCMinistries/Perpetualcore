Output: Updated draft/redraft/review routes, each gated and single-counting.
</objective>

  <action>
For BOTH routes (draft and redraft — they share the drafter_v1 / generateDraft pattern):

1. Import `{ guardedLLMCall, budgetExceededResponse, BudgetExceededError }` from `@/lib/rfp/ai/guardrail`.

2. Resolve the org_id the route already has (draft route: `body.org_id`; redraft: whatever org it resolves from the proposal — read the file to confirm the variable). The wrapper needs the tenant's org id.

3. Wrap the existing `generateDraft({...})` call in `guardedLLMCall`. The fn() must return an object satisfying `LLMCallMeta` — generateDraft already returns `{ tokens_in, tokens_out, cost_usd, model, session_id, sections, ... }`. Adapt by returning from fn() the draft object PLUS the LLMCallMeta-shaped fields the wrapper reads: map `agent: "drafter_v1"`, `model: draft.model`, `tokensIn: draft.tokens_in`, `tokensOut: draft.tokens_out`, `costUsd: draft.cost_usd`, `sessionId: <the sessionIdWithFlags value currently built>`, `proposalId: proposal.id`. Two clean options — pick the cleaner for each file:
   (a) Have fn() call generateDraft, then build and return `{ ...draft, agent, model, tokensIn, tokensOut, costUsd, sessionId, proposalId }`. NOTE the draft route currently inserts the proposal/sections AFTER generateDraft and only then knows proposal.id; in that case wrap ONLY the generateDraft LLM call in guardedLLMCall so the budget check happens before the model fires, and pass proposalId/sessionId via a post-call recordCost is NOT desired (would double the work). Simpler: wrap generateDraft so the PRE-CALL budget check runs, return draft+meta WITHOUT proposalId (proposalId is unknown pre-insert), and let the wrapper record the session row. Set proposalId by performing the proposal insert FIRST is invasive — instead omit proposalId (nullable in the session row) OR move the session attribution to include proposal_id by having the wrapper accept it. Choose: omit proposalId in the meta (the wrapper inserts proposal_id: null), which is acceptable for cost ledgering (org_id is what budget needs). Document this minor change (draft session rows lose proposal_id linkage) in the SUMMARY.
   Cleaner alternative if proposalId linkage matters: keep the proposal/section inserts where they are, and have the wrapper's fn() return meta WITHOUT proposalId; if you want proposal_id on the ledger row, that's a known minor tradeoff — note it. Do NOT restructure the whole route to preserve it.

4. CRITICAL — remove the inline `await admin.from("rfp_agent_sessions").insert({...})` block (draft route lines ~340-349 build sessionIdWithFlags then insert). The wrapper's recordCost now writes that row. Keep building any session_id flag string ONLY if you pass it to the wrapper as `sessionId`; otherwise delete it. The end state: exactly ONE rfp_agent_sessions insert per call, performed by the wrapper.

5. Wrap the call in try/catch; on `err instanceof BudgetExceededError` return `budgetExceededResponse(err)`. Preserve all existing non-budget error handling (generation_failed 502, etc.) unchanged.

6. The budget check must happen BEFORE generateDraft fires. Because guardedLLMCall runs checkBudget() before fn(), placing generateDraft inside fn() guarantees this — verify generateDraft is the ONLY thing inside fn() that calls the model.

Surgical: do not touch auth, membership checks, alias resolution, vault retrieval, or proposal/section insert logic beyond removing the inline session insert.

Manual trace: an over-budget org throws BudgetExceededError from checkBudget before generateDraft → route returns 402.</verify>
